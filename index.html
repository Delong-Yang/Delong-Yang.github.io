Spring 事务传播机制

Posted on 2019-06-25

Spring 支持两种事务管理方式， 声明式 和编程式

编程式比较直观，程序员通过TractionTemplate来控制事务的提交、回滚。但对于业务代码的侵入性较大。

声明式通过AOP来实现，可以通过xml配置或者@Transtional注解声明事务, 使用起来简单方便，对于业务代码没有影响。

Spring默认提供的7种事务传播方式来控制什么情况下需要提交/回滚事务， 尤其实在是在事务有嵌套情况下理解起来并不是很容易。下面我们通过几个小例子来观察一下程序的行为， 我们在数据库中创建了两张表emp和mgr:

单一事务的行为

Spring默认在发生RuntimeException时会回滚事务， checked Exception则不会回滚。

可以通过@Transactional的rollbackFor和noRollbackFor来修改默认行为。需要注意的是，该设置同样适用于被指定的类的子类。

事务传播的类型

@Transactional中还有一个属性：Propagation， Propagation 是一个枚举，一共有7种类型：

REQUIRED

如果当前线程中已有一个事务，则继续在当前事务中执行。否则创建一个新的事务。

SUPPORTS

如果当前线程中已有一个事务，则继续在当前事务中执行。否则以非事务形式执行。

MANDATORY

如果当前线程中已有一个事务，则继续在当前事务中执行。否则抛出异常。

REQUIRES_NEW

如果当前线程中已有一个事务，则挂起当前事务， 并在新创建的事务中执行。

NOT_SUPPORTED

以非事务形式执行， 如果当前线程中已有一个事务，则挂起该事务

NEVER

以非事务形式执行，如果当前线程已有事务，则抛出异常

NESTED

如果当前线程中已有一个事务，则在一个嵌套的事务中执行， 否则穿件一个新的事务

其中4种比较容易理解， REQUIRED REQUIRES_NEW NESTED 比较容易混淆。 我们通过几个例子来看一起看一下。

示例

这里创建两个Service: EmpService和SalaryService, 在EmpService中创建了一条记录之后调用SalaryService插入薪水, 并手工抛出异常来触发回滚。

  1. REQUIRED 是默认的传播行为， SalaryService 会在EmpService创建的事务中执行，当SalaryService触发回滚时，会连同EmpService整体作为一个Transaction回滚。数据库中并没有数据插入。

  1. 把SalaryService 中的传播方式改REQUIRES_NEW 会挂起当前事务Tx1，并新建一个Tx2去执行SalaryService中的操作，当SalaryService发生异常时只会回滚Tx2。 Tx1 中Emp数据成功插入。

  1. 把SalaryService的传播方式改为NESTED后，观测到和REQUIRED一样的，Emp提交成功， Salary被回滚。

About-Kerberos-Principal-And-Key

Posted on 2017-08-24

The original English blog is from https://ssimo.org/blog/id_016.html

Kerberos Principal是什么？

简单来说principal作用等同用user name。Kerberos Princial 分为User Principal 和Service Princial。单纯只用username并不能很好的区分这两者的概念，因此Principal也就应运而生了。

Principal 长啥样？

一个Principal由几个字符串组成，@后面的是Realm， 习惯上全部使用大写字母。@前面是Realm中的一个限定符(Identity), 可能由几个部分组成，用”/“分隔。

Example:

compoennt1/component2@REALM

User Pricipal只有一个component和Realm, 比如*delong@EXAMPLE.COM*表示一个叫delong的用户属于EXAMPLE.COM这个Realm。
在AD(Active Directory)中，Realm就是domain name.

Service Principal有两个compoments，service和hostname

Example:

HTTP/server.example.com@EXAMPLE.COM

当client向KDC(Key distribution centre)请求访问service的ticket时，就要用到principal。

Keys and Keytabs

每个princial在KDC中都关联有一个key,用来加密发送给client的ticket，Service需要用相同的key解密ticket。所以Kerberos也称为shared key system。

User principal的key是user的password。 KDC中也存有user的password。

Service principal 就不用密码了，一般是随机生成的key，一份存放在KDC，一份存放在keytab文件里。

Create service principal name On the Windows Active Domain Controller:

$ setspn -A HTTP/server.example.com serviceADUser

To create the keytab file use the following process:

$ ktpass -out c:\dir\krb5.keytab -princ HTTP/server.example.com@EXAMPLE.COM -mapUser serviceADUser -mapOp set -pass serviceADUserPWD -crypto RC4-HMAC-NT -pType KRB5_NT_PRINCIPAL
